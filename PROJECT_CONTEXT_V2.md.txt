🛠️ Project Overview & Live Environment
Voyage (~/travel-planner) is an intelligent travel orchestration web app that integrates logistics (flights, lodging, transit) with an automated, constraint-aware itinerary generation engine.

Core Stack: Next.js 15, React 19, Supabase (Auth + Postgres + RLS), Tailwind v4, shadcn/ui, @dnd-kit.

Active Test Asset: Barcelona 2026 | Trip ID: 8ed96750-9eaa-4f81-9967-72da883cae74.

Trip Scope: Recently adjusted from 15 days down to 12 days (Jul 1–12, 2026).

Local Address: http://localhost:3000 (Fallback to 3002/3003 if ports are ghosted by stale tasks).

🧭 Unbreakable Product & Design Rules
Manual Places Guarantee: Every venue with source = 'manual' (15 spots in Barcelona, including Sagrada Família and pinned reservations) must appear exactly once. They are mandatory anchors and cannot be dropped or substituted for suggestions.

Sacred Reservation Anchors: User-pinned reservation times are absolute. The time-chaining travel engine must never shift or travel-chain over them (e.g., Icebarcelona at 8:45 PM on Jul 3).

Strict Global Deduplication: Venues are deduplicated trip-wide via their unique google_place_id. If a conflict occurs between a manual user addition and a generator suggestion, the manual record wins, and the duplicate suggestion is evicted.

Operational Hour Isolation: Events must respect cached time limits. Meals cannot land before opening hours (e.g., Soma cannot be at 8:19 AM if it opens at 9:00 AM).

Strict Category Constraints:

Outdoor Parks/Nature/Hikes: Must conclude operations by 6:00 PM (preventing late-night trail walking).

Experiences/Guided Tours: Must conclude operations by 6:30 PM.

Nightlife/Bars: Enforced time-floors prevent them from slotting into morning hours.

UX Responsiveness: Full-screen blocking spinners are exclusive to the master Generate action. Minor adjustments (drag-and-drop reordering, travel time fetching via directionsLoading) use per-day indicators and non-blocking background states.

Map Pool Separation:

My Map / My Places: Displays only manual user selections (source === 'manual').

Plan Map: Dynamically tracks only active generated itinerary steps for the selected day.

🚀 Recent Session Accomplishments & Engineering Fixes
1. Architectural Overhauls & Scheduling Passes
Two-Pass Pipeline Sync: To prevent travel calculations from accidentally overwriting opening-hour layouts, the core execution engine now uses shared time verification (resolveMealArrivalMinutes and resolveVisitArrivalMinutes) across both the placement pass and the recompute/reschedule pathways (schedule-times.ts + reschedule-day.ts).

Synchronous Enrichment: Integrated enrichPlacesOpeningHours into the live pipeline before reschedules map out. This guarantees operating hour data is pulled immediately before computing dates rather than lazily in the background.

Sit-Down Separation: Introduced isSitDownRestaurant() to parse out and exclude tasting classes, tours, or markets from landing inside core breakfast/lunch/dinner structural slots.

2. UI Features Launched
Plan Map Day Filters: Selecting a day card now filters active plan pins specifically to that day’s routing.

Trip Date Editing & Sync: Integrated a master trip editing sheet (trip-dates-dialog.tsx) to alter active dates, sync itinerary_days, and automatically purge out-of-range rows.

Completion Flags: Implemented migration 006, adding an is_completed boolean toggle to itinerary_stops. Once checked, the system permanently filters that venue out of future suggestion top-ups.

3. Edge-Case Bug Patches Resolved in Code
The Sagrada/Park Güell Duplication: Fixed an exception where the manual-places algorithm skipped user entries if the suggestion pool had already grabbed that google_place_id. Added dedupeStopsPreferManual and evictSuggestedGoogleDuplicate to force manual retention.

Tour Timing Flaws: Added regex word-boundary flags (\btours?\b) to isExperienceActivity to catch unclassified text descriptions like "BARKENO TOURS" and clip them at the 6:30 PM operational wall.

Nightlife Engine Refusal: Patched a fallback error where resolveVisitArrivalMinutes returned null for late-night venues (e.g., Gatsby), which historically caused the recalculation engine to leave the arrival chain frozen at morning values (8:24 AM).

## Key decisions — do not regress

1. Day rhythm is mandatory.

   * Sort and chain stops by phase.
   * Do not rely on raw clock order or stale `sort_order`.

2. Reservation times are anchors.

   * Never travel-chain over reservation times.
   * Reserved attractions/restaurants must keep their sacred time.

3. Breakfast must respect opening hours.

   * Default breakfast floor is 9:00 AM when opening hours are unknown.
   * Implemented through `getMealEarliestMinutes`.

4. Breakfast can become brunch.

   * If no early-open breakfast fits before a morning anchor like Casa Batlló, schedule brunch after the visit, around 10:15 AM or later.

5. Three meals every day.

   * `ensureTripMeals` is the safety net after generation.
   * It must loop through all `itinerary_days`.

6. Reserved restaurants need meal labels.

   * Example: Cañete at noon should have `meal_type: lunch`.
   * UI should show a badge when `meal_type` is set, not only when `stop_type === "meal"`.

7. Two-pass timing matters.

   * Placement fixes must also hit `recomputeScheduleTimes` / `rescheduleStopsFromOrder`.

8. Regenerate is required.

   * Existing `itinerary_stops` are not automatically migrated.
   * The current DB itinerary should be considered stale until a full rebuild/regenerate runs.
